# 🌌 Phase 11: RAG & Audit Engine Accuracy, Coverage, and Performance - 2026-06-17

## Summary
Completed all Phase 11 tasks in `PLANNER.md`. Found and fixed two real, previously-silent bugs: (1) `mapErrorToHdcAiTerm()` in `validator.js` never mapped the REQ ("ขาดข้อมูลบังคับ") or FMT ("รูปแบบไม่ถูกต้อง") error codes — 2 of the 5 canonical `MKB.ERRORS` codes fell through unmapped, and a loose `"CID"` substring check could misclassify FMT-with-`(CID)`-suffix messages as CRC errors; (2) the Phase 9 self-learning RAG cache lookup read a `localStorage` key that `initRagCacheDB()` permanently deletes after one-time migration to IndexedDB, making that cache a silent no-op for any returning user since it shipped. Also capped previously-unbounded per-table prompt blocks (was scaling with all ~19 `TABLE_KPI_MAP` entries matched) and removed a redundant `.toLowerCase()` re-scan of the ~1,812-chunk RAG corpus on every query. Audited all 86 files under `RAG/` and confirmed the knowledge base has no missing ingestable content — the only gaps are intentionally-excluded developer files and one image-only PPTX needing OCR (out of scope). Added 3 new regression tests under `tests/`; all pass.

## Errors Solved
- `mapErrorToHdcAiTerm` silently passed through REQ ("ขาดข้อมูลบังคับ") and FMT ("รูปแบบไม่ถูกต้อง") error messages unmapped, and could misclassify an FMT message with a `(CID)` field suffix as a CRC/checksum error.
- The Phase 9 self-learning RAG cache (`queryLocalRAG`'s "1. Check local self-learning cache" section) was reading from a `localStorage` key that gets deleted right after the one-time IndexedDB migration and is never repopulated — making this feature permanently inert since it shipped.
- Unbounded per-table prompt blocks: a broad query matching many of the ~19 `TABLE_KPI_MAP` tables could inject a KPI block and an error-correlation block per match with no cap, unlike the RAG chunk matcher which was already capped at top 3.

## Next Steps
1. If OCR support is ever wanted for image-only slide decks (e.g. the NCDSCREEN PPTX found during this audit), that's a new dependency/capability (e.g. `pytesseract`) and should be scoped as its own task rather than folded into `rag_builder.py` silently.
2. Consider running `rag_builder.py` for real the next time a genuinely new text-bearing manual is added to `RAG/` — the script change in this report only adds visibility into skipped files, it does not need to be re-run right now since no new ingestable content exists.
3. No deploy step was run as part of this report (no `validator.js`-affecting user-facing regression expected, but recommend a quick manual smoke test of the chatbot before the next push, per the project's established convention in `REPORT-2026-06-14.md`).
